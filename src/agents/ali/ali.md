# 笔试题解析

## 题目来源

- 技术方向：计算机基础 / 前端开发

## 题目整理

### 第 1 题 [单选题]

#### 题目

某开发者为解决一个复杂问题编写了一个计算斐波那契数列的递归函数。如果她传入的数值非常大，导致递归调用链过长，最有可能直接引发的运行时错误是（）

#### 选项

- A. 堆溢出（Heap Overflow）
- B. 栈溢出（Stack Overflow）
- C. 内存泄漏（Memory Leak）
- D. 空指针解引用（Null Pointer Dereference）

#### 答案

B. 栈溢出（Stack Overflow）

#### 解析

递归函数每深入一层调用，都会在调用栈上新增一个栈帧，用于保存函数参数、返回地址和局部变量等信息。计算斐波那契数列的朴素递归实现会产生非常深、非常多的函数调用；当输入值很大时，递归深度可能超过线程栈的容量限制，最终触发栈溢出。

其他选项不符合这里的直接原因：

- A. 堆溢出通常与动态内存分配过多有关，而不是递归调用层数本身。
- C. 内存泄漏是申请的内存未被正确释放，重点在生命周期管理问题，不是本题描述的主要风险。
- D. 空指针解引用发生在访问无效指针时，而题目强调的是递归过深。

#### 延伸说明

如果要避免这类问题，常见思路包括：

- 将递归改写为迭代
- 使用动态规划或记忆化搜索减少重复计算
- 在工程实践中避免对极大输入直接使用朴素递归实现

### 第 2 题 [单选题]

#### 题目

关于 SFT 与 DPO/RLHF 类偏好对齐的差异，哪项表述更准确？（）

#### 选项

- A. SFT 用成对偏好数据训练；对齐用单条监督数据训练
- B. SFT 主要做监督模仿；对齐利用偏好信号优化输出
- C. SFT 只调 embedding；对齐只调 attention
- D. SFT 只改解码策略；对齐不改模型参数

#### 答案

B. SFT 主要做监督模仿；对齐利用偏好信号优化输出

#### 解析

`SFT`（Supervised Fine-Tuning，监督微调）的核心是利用高质量指令数据或标注答案做监督学习，让模型学习“在给定输入下应该如何作答”。它更接近模仿学习，本质上是让模型逼近示范答案的分布。

`DPO` 或 `RLHF` 这类偏好对齐方法则更强调“输出偏好优化”。它们不是简单复现单条标准答案，而是根据“哪个回答更好”的偏好信号，推动模型更稳定地生成更符合人类偏好的结果，例如更有帮助、更安全、更符合风格要求。

其他选项的问题在于：

- A. 把两类方法的数据形式说反了。通常 SFT 使用单条监督样本，而偏好对齐更常使用成对偏好数据或奖励信号。
- C. 这两类方法都不局限于只训练某一个特定模块，是否只训练部分参数取决于具体训练配置，而不是方法定义本身。
- D. SFT 不是只改解码策略，而是通过训练更新模型参数；DPO/RLHF 也通常会影响模型参数，而不是完全不改参数。

#### 延伸说明

在大模型训练流程里，常见顺序通常是：

- 先做预训练，获得通用语言能力
- 再做 SFT，让模型学会遵循指令和模仿高质量回答
- 最后做 DPO 或 RLHF 等偏好对齐，使输出更符合人类偏好与产品目标

### 第 3 题 [单选题]

#### 题目

在使用快速排序对大规模数据进行排序时，若每次都选择待排序区间的第一个元素作为主元（pivot），在何种输入数据下，其性能最可能退化到最差的 O(n^2) 级别？（）

#### 选项

- A. 数据完全随机分布
- B. 数据中包含大量重复元素
- C. 数据已经完全有序或完全逆序
- D. 数据呈正态分布

#### 答案

C. 数据已经完全有序或完全逆序

#### 解析

快速排序的平均时间复杂度是 `O(n log n)`，前提是每次划分都相对均衡。如果每次都固定选择区间第一个元素作为主元，那么当输入数据已经完全有序或完全逆序时，这个主元往往会成为当前区间的最小值或最大值。

这样一来，每一轮划分都会出现极端不平衡的结果：一边子区间规模为 `0`，另一边子区间规模为 `n-1`。递归关系就会退化为：

`T(n) = T(n-1) + O(n)`

最终总体时间复杂度退化为 `O(n^2)`。

其他选项不够准确：

- A. 随机分布通常更接近快速排序的平均情况，不容易持续产生最坏划分。
- B. 大量重复元素在某些实现下可能影响性能，但题目强调的是“固定选第一个元素”时最典型、最直接的最坏情况。
- D. 正态分布并不会天然导致每次划分都极不平衡。

#### 延伸说明

工程上为了避免这种最坏情况，常见优化方式包括：

- 随机选择主元
- 使用三数取中法选择主元
- 对大量重复元素使用三路快排
- 当递归过深时切换到堆排序等保底策略

### 第 4 题 [单选题]

#### 题目

一家电影院需要分析影片放映情况。已知数据库包含如下两张表：

- `movies`（影片表）
  - `movie_id`：影片标识符
  - `movie_title`：影片名称
  - `genre`：类型
  - `duration_min`：时长（分钟）
  - `release_date`：上映日期
- `screenings`（放映记录表）
  - `screening_id`：放映标识符
  - `movie_id`：影片标识符
  - `screening_time`：放映时间
  - `hall_number`：影厅编号
  - `tickets_sold`：售出票数

需要查询“在多个影厅放映过的影片名称及放映影厅数量”，以下 SQL 语句正确的是（）

#### 选项

- A. `SELECT movie_title, COUNT(DISTINCT hall_number) AS hall_count FROM movies m JOIN screenings s ON m.movie_id = s.movie_id GROUP BY movie_title HAVING COUNT(DISTINCT hall_number) > 1`
- B. `SELECT movie_title, COUNT(hall_number) AS hall_count FROM movies m JOIN screenings s ON m.movie_id = s.movie_id GROUP BY movie_title HAVING COUNT(hall_number) > 1`
- C. `SELECT movie_title, COUNT(DISTINCT hall_number) AS hall_count FROM movies m JOIN screenings s ON m.movie_id = s.movie_id WHERE COUNT(DISTINCT hall_number) > 1 GROUP BY movie_title`
- D. `SELECT DISTINCT movie_title, hall_number FROM movies m JOIN screenings s ON m.movie_id = s.movie_id HAVING COUNT(*) > 1`

#### 答案

A. `SELECT movie_title, COUNT(DISTINCT hall_number) AS hall_count FROM movies m JOIN screenings s ON m.movie_id = s.movie_id GROUP BY movie_title HAVING COUNT(DISTINCT hall_number) > 1`

#### 解析

题目要求的是：

- 按影片统计
- 统计该影片一共在多少个不同影厅放映过
- 只保留“影厅数量大于 1”的影片

因此正确思路必须同时满足三点：

- 通过 `JOIN` 将影片表和放映记录表按 `movie_id` 关联
- 使用 `GROUP BY movie_title` 按影片分组
- 使用 `COUNT(DISTINCT hall_number)` 统计“不同影厅数量”，并通过 `HAVING ... > 1` 过滤出在多个影厅放映过的影片

选项 A 完整满足上述要求，因此是正确答案。

其他选项的问题如下：

- B. 使用的是 `COUNT(hall_number)`，统计的是放映记录条数，而不是不同影厅数量。如果同一影片在同一影厅放映多次，会被重复计数。
- C. 把聚合条件写在 `WHERE` 中是错误的。`WHERE` 不能直接使用聚合函数，分组后的过滤应放在 `HAVING` 中。
- D. 既没有正确地按影片聚合统计不同影厅数量，也没有形成题目要求的结果结构；`HAVING COUNT(*) > 1` 在这里的使用方式也不成立。

#### 延伸说明

这道题考查的是 SQL 中几个非常基础但高频的点：

- 明确“统计记录数”与“统计去重后的类别数”之间的区别
- 理解 `WHERE` 与 `HAVING` 的适用阶段不同
- 涉及“每个对象满足某种聚合条件”的查询时，通常要考虑 `GROUP BY + 聚合函数 + HAVING`

### 第 5 题 [单选题]

#### 题目

以下关于红黑树（Red-Black Tree）性质的描述中，哪一项是错误的（）

#### 选项

- A. 从根到任意一个 `NIL`（叶子）节点的每条路径都包含相同数目的黑色节点
- B. 如果一个节点是红色的，则它的两个子节点都是黑色的
- C. 根节点必须是黑色的
- D. 任何一个节点到其所有后代 `NIL` 节点的简单路径上，黑色节点的数量可以不相同

#### 答案

D. 任何一个节点到其所有后代 `NIL` 节点的简单路径上，黑色节点的数量可以不相同

#### 解析

红黑树的核心约束之一就是黑高（black height）一致性：对于任意一个节点，从该节点到所有后代 `NIL` 叶子节点的简单路径上，都必须包含相同数量的黑色节点。这个性质保证了红黑树不会退化得过于严重，从而使查找、插入、删除等操作能够保持 `O(log n)` 的时间复杂度。

因此，选项 D 说“黑色节点数量可以不相同”，这与红黑树的定义直接矛盾，所以它是错误表述。

其他选项都是标准性质：

- A. 这是黑高一致性的常见表述方式，从根到各个 `NIL` 叶子的黑节点数相同。
- B. 红节点不能有红色子节点，也就是不存在相邻的两个红节点。
- C. 在标准定义中，根节点为黑色。

#### 延伸说明

红黑树通常满足以下几条经典性质：

- 每个节点不是红色就是黑色
- 根节点是黑色
- 所有 `NIL` 叶子节点视为黑色
- 红色节点的子节点必须是黑色
- 从任意节点到其所有后代 `NIL` 叶子节点的路径上，黑节点数量相同

正是这些约束共同保证了红黑树是一种“近似平衡”的二叉搜索树。

### 第 6 题 [单选题]

#### 题目

关于 HTTP 缓存再验证（conditional request），更符合规范的做法是？（）

#### 选项

- A. 服务端下发 `ETag`，客户端用 `If-None-Match` 发起条件请求
- B. 服务端下发 `Cookie`，客户端用 `If-Modified-Since` 发起条件请求
- C. 服务端下发 `Cache-Control: no-store`，客户端用 `If-Match` 命中本地缓存
- D. 服务端下发 `Content-Type`，客户端用 `If-Range` 发起条件请求

#### 答案

A. 服务端下发 `ETag`，客户端用 `If-None-Match` 发起条件请求

#### 解析

HTTP 条件请求的核心思路是：客户端先缓存资源的某个“验证器”，下一次请求时把这个验证器带给服务端，由服务端判断资源是否发生变化。如果没有变化，服务端返回 `304 Not Modified`，客户端继续使用本地缓存。

在实践中，最典型、最规范的一组搭配就是：

- 服务端响应头下发 `ETag`
- 客户端后续请求头带上 `If-None-Match`

服务端比较当前资源的 `ETag` 与客户端提交的值：

- 如果一致，说明资源未变化，可返回 `304`
- 如果不一致，说明资源已变化，返回新的资源内容和新的 `ETag`

因此选项 A 是正确答案。

其他选项的问题如下：

- B. `Cookie` 不是缓存再验证用的资源验证器，和 `If-Modified-Since` 并不是一组对应的缓存机制。
- C. `Cache-Control: no-store` 的语义是“不允许存储”，既然都不允许缓存，就谈不上后续基于本地缓存做再验证；而 `If-Match` 主要用于并发控制、更新前校验等场景，也不是这里最典型的缓存再验证做法。
- D. `Content-Type` 描述的是资源媒体类型，不是缓存验证器；`If-Range` 主要用于范围请求（Range Request）相关场景，不是本题所问的常规缓存再验证机制。

#### 延伸说明

HTTP 缓存再验证常见有两类验证器：

- 基于时间：`Last-Modified` + `If-Modified-Since`
- 基于实体标识：`ETag` + `If-None-Match`

其中 `ETag` 一般更精确，因为它不只依赖修改时间，还可以基于资源内容或版本号进行判断。

### 第 7 题 [多选题]

#### 题目

关于 Transformer 训练中的混合精度与数值稳定性，以下哪些说法是正确的？

#### 选项

- A. 混合精度训练常用 loss scaling 来降低下溢风险
- B. 使用 FP16/BF16 通常能减少显存占用并提高吞吐
- C. BF16 相比 FP16 通常有更大的指数范围，训练更稳定
- D. 混合精度会改变模型结构，因此会影响推理时架构兼容性

#### 答案

A、B、C

#### 解析

这道题考查的是混合精度训练的基本机制与工程效果。

- A 正确。混合精度训练中，尤其是在使用 FP16 时，数值表示范围较小，梯度容易出现下溢。`loss scaling` 的作用就是在反向传播前把 loss 放大，从而让梯度值落在更容易表示的范围内，减少下溢风险。
- B 正确。使用 FP16 或 BF16 进行训练，通常会降低激活值、梯度和部分中间张量的存储成本，因此可以减少显存占用；同时现代硬件通常对半精度计算有更高吞吐，因此训练速度也往往更快。
- C 正确。BF16 与 FP16 相比，尾数位更少，但指数位更多，因此动态范围更接近 FP32。在很多大模型训练场景下，BF16 更不容易出现溢出或下溢，整体训练稳定性通常优于 FP16。
- D 错误。混合精度主要改变的是训练或推理时的数据表示精度，以及部分算子的执行精度策略，并不会改变模型的网络结构本身。因此它不会因为“修改模型结构”而影响架构兼容性。

#### 延伸说明

工程实践里，混合精度训练通常会结合以下策略：

- 对部分敏感操作保留 FP32 精度
- 在 FP16 训练中使用动态或静态 `loss scaling`
- 在支持良好的硬件上优先考虑 BF16，以兼顾吞吐和稳定性

因此，混合精度的关键并不是“改模型结构”，而是在性能收益与数值稳定性之间取得更好的平衡。

### 第 8 题 [多选题]

#### 题目

大促期间，阿里巴巴数据平台需要实时维护每个商品类目下销量排名前 100 的商品列表。研发团队在进行内存与时间复杂度权衡后，选择优先队列（底层基于堆结构）来解决这类 Top-K 流数据处理问题。下列关于该数据结构选择与性能特征的叙述合理的有（）

#### 选项

- A. 维护大小固定为 100 的小顶堆能以较低的时间开销过滤小流量商品
- B. 每次新商品销量更新导致堆节点调整的时间开销复杂度为对数级别
- C. 采用大顶堆结构可以进一步将新元素的插入时间复杂度降低至常数级
- D. 当流数据规模达到数亿级别时，该堆的系统内存占用率会随之呈暴增

#### 答案

A、B

#### 解析

这道题的关键在于理解 Top-K 流式处理里为什么通常使用“固定容量的小顶堆”。

对于“维护销量前 100 的商品”这个目标，常见做法是：

- 使用一个大小固定为 `100` 的小顶堆
- 堆顶保存当前 Top-100 里最小的那个元素
- 新元素到来时，先与堆顶比较
- 如果新元素更大，就替换堆顶并做堆调整；否则直接丢弃

基于这个思路：

- A 正确。固定大小的小顶堆非常适合过滤大量“不可能进入前 100”的低销量商品，因为大多数元素只需要和堆顶比较一次就能被淘汰，整体时间和空间开销都较低。
- B 正确。堆插入、删除堆顶、替换后下沉或上浮等操作，时间复杂度通常是 `O(log K)`。本题中 `K = 100`，因此每次调整的复杂度是对数级别。
- C 错误。改成大顶堆并不能把插入复杂度降到常数级。堆的结构性调整本质上仍然需要 `O(log K)`。而且在 Top-K 最大值问题中，大顶堆通常不如固定容量小顶堆直接高效。
- D 错误。该方法的关键优势之一恰恰是空间复杂度只与 `K` 有关，而不是与总流数据规模线性相关。即使流数据达到数亿级，只要只维护前 100，堆本身的核心存储规模仍然是 `O(K)`，不会随着总数据量暴涨。

#### 延伸说明

Top-K 场景中常见复杂度结论是：

- 时间复杂度：整体通常可视为 `O(N log K)`
- 空间复杂度：`O(K)`

当 `K` 远小于 `N` 时，这种方案比对全量数据排序更加适合实时流式场景。

#### 识别备注

从题干“叙述合理的有”这一措辞判断，本题更像多选题；如果原系统界面明确标注为单选，则需要以出题系统定义为准后再复核。

### 第 9 题 [多选题]

#### 题目

关于快速重传与快速恢复（以 Reno 语义为例），哪些说法较准确？

#### 选项

- A. 重复 ACK 达到一定阈值后可能触发快速重传
- B. 快速重传用于在 RTO 到期前更早地重传疑似丢失段
- C. 快速恢复阶段会调整 `cwnd` 并尝试维持一定发送速率
- D. 快速重传发生后通常需要重新进行三次握手

#### 答案

A、B、C

#### 解析

这道题考查的是 TCP Reno 中“快速重传”和“快速恢复”的核心机制。

- A 正确。在 Reno 中，当发送方连续收到一定数量的重复 ACK（经典实现里通常是 3 个重复 ACK）时，会推断某个报文段很可能已经丢失，从而触发快速重传。
- B 正确。快速重传的意义就在于不必等到重传超时 `RTO` 到期，再去补发丢失报文；它能更早地根据重复 ACK 信号进行重传，从而缩短恢复时间。
- C 正确。快速恢复的目标是避免每次丢包都把发送端完全打回慢启动起点。它会调整拥塞窗口 `cwnd` 和相关阈值，使连接在检测丢包后仍尽量维持一定发送能力，而不是完全停顿。
- D 错误。快速重传只是 TCP 连接内部的数据重传与拥塞控制行为，并不意味着连接断开，因此通常不需要重新进行三次握手。三次握手发生在建立新连接时，而不是普通丢包恢复时。

#### 延伸说明

Reno 语义下的一个常见流程是：

- 收到多个重复 ACK，触发快速重传
- 进入快速恢复，调整 `ssthresh` 和 `cwnd`
- 在恢复阶段尽量保持链路利用率
- 等到确认恢复后，再回到较正常的拥塞控制节奏

这也是为什么快速重传与快速恢复通常是配套理解的：前者解决“更早发现并重传丢包”，后者解决“发生丢包后不要让吞吐掉得过猛”。

- 模式-内模式映射：支持物理独立性

这也是数据库系统能够在“用户视图、逻辑设计、物理存储”三个层次上进行解耦的重要基础。

### 第 11 题 [多选题]

#### 题目

在操作系统的并发编程中，死锁是多线程程序常见的问题之一。某系统中有两个线程和两把互斥锁 A、B，线程 1 先获取锁 A 再获取锁 B，线程 2 先获取锁 B 再获取锁 A。关于该场景下的死锁问题，下列说法正确的有（）

#### 选项

- A. 当线程 1 持有锁 A 等待锁 B，同时线程 2 持有锁 B 等待锁 A 时，就会发生死锁
- B. 规定所有线程必须按照相同的顺序获取锁，可以预防此类死锁
- C. 死锁一旦发生，只能通过终止进程解除，没有其他办法
- D. 该场景满足死锁的四个必要条件中的循环等待条件

#### 答案

A、B、D

#### 解析

这道题考查的是死锁形成条件以及常见预防方法。

- A 正确。如果线程 1 已持有锁 A 并等待锁 B，线程 2 已持有锁 B 并等待锁 A，那么双方都在等待对方释放资源，彼此无法继续推进，这就是典型死锁场景。
- B 正确。统一加锁顺序是预防死锁的经典方法。例如规定所有线程都必须先获取锁 A，再获取锁 B，就可以破坏循环等待条件，从而避免这类死锁。
- C 错误。死锁发生后，终止进程只是解除死锁的一种粗暴方式，并不是唯一办法。实际系统里还可能采用超时回退、死锁检测后回滚、资源抢占、事务中止等机制处理。
- D 正确。线程 1 等待线程 2 释放锁 B，线程 2 等待线程 1 释放锁 A，形成了一个等待环，因此满足死锁四个必要条件中的循环等待条件。

#### 延伸说明

经典的死锁四个必要条件包括：

- 互斥
- 占有并等待
- 不可抢占
- 循环等待

在工程实践中，最常见的预防策略之一就是：

- 统一资源获取顺序

因为这可以直接破坏“循环等待”这一必要条件，通常实现简单且效果明显。

### 第 12 题 [多选题]

#### 题目

在电商分布式库存系统中，某单调递增的商品库存分片序列因集群扩扩容发生了一次未知的循环位移，例如原有序列 `[0,1,2,4,5,6,7]` 可能变为了 `[4,5,6,7,0,1,2]`。现系统需要在改造后的序列中查找是否存在目标库存分片标识。若为了保证高并发检索性能，强制采用时间开销为对数级的算法完成查找，下列关于该算法设计的表述合理的有（）

#### 选项

- A. 算法的核心思路依然是利用分治思想通过中间节点逐步缩小查找区间
- B. 每次将区间一分为二后必然能保证至少有一半的区间处于完全有序状态
- C. 即便序列中存在海量重复元素也能始终稳定保持对数级的查找开销
- D. 需要通过比较中间元素与边界元素的大小关系来决定目标可能落在的区间

#### 答案

A、B、D

#### 解析

这道题本质上考查的是“旋转有序数组查找”这一经典问题。它通常仍然基于二分查找思想，只是判断逻辑比普通有序数组更复杂。

- A 正确。核心思路仍然是分治和二分，每次通过中间位置 `mid` 缩小搜索区间，而不是线性扫描。
- B 正确。在不考虑重复元素干扰的标准旋转有序数组场景下，任意时刻总能判断出左半段或右半段至少有一段保持有序，因此可以据此继续缩小范围。
- C 错误。如果数组中存在大量重复元素，例如边界值和中间值频繁相等，那么有时无法有效判断哪一侧严格有序，算法可能退化到接近线性扫描，不能保证“始终稳定保持对数级”。
- D 正确。判断目标值应该落在左半区还是右半区，通常需要比较 `mid` 与左右边界元素的大小关系，再结合目标值范围决定下一步搜索方向。

#### 延伸说明

标准旋转有序数组查找的一般逻辑是：

- 取中间位置 `mid`
- 判断左半区是否有序，或右半区是否有序
- 如果目标值落在有序区间内，就进入该区间继续二分
- 否则进入另一侧继续搜索

因此它相比普通二分查找，多出的关键在于：

- 识别哪一半仍保持有序
- 根据目标值与边界值的关系决定搜索方向

#### 识别备注

本题在常规“无大量重复元素干扰”的旋转有序数组查找前提下，`A、B、D` 是合理答案；若题目特别强调“允许大量重复元素且仍要求严格对数级保证”，则需要额外指出这一要求本身在理论上并不总是可满足。

Node.js 服务中短生命周期对象创建过多导致 GC 压力上升，较常见的改进方式是？

#### 选项

- A. 复用可复用对象或使用对象池降低分配频率

#### 延伸说明

Node.js 中减少 GC 压力的常见方式包括：

- 避免无意义的中间对象创建
- 复用 Buffer、数组或对象
- 尽量缩短对象存活时间，避免意外提升到老生代
- 结合性能分析工具定位真实分配热点

### 第 14 题 [单选题]

#### 题目

阅读代码，选择最终输出结果：

```javascript
(() => {
  function Foo() {}
  Foo.prototype.x = 1;
  const a = new Foo();
  Foo.prototype = { x: 2 };
  const b = new Foo();
  console.log(a.x + "|" + b.x);
})();
```

#### 选项

- A. `1|2`
- B. `2|2`
- C. `1|1`
- D. `undefined|2`

#### 答案

A. `1|2`

#### 解析

`a` 在创建时，其原型链已经指向当时的 `Foo.prototype`，其中 `x = 1`。后面执行：

```javascript
Foo.prototype = { x: 2 };
```

这一步不是修改旧原型对象上的属性，而是让 `Foo.prototype` 指向了一个全新的对象。因此：

- `a` 仍然沿着旧原型对象查找，得到 `a.x === 1`
- `b` 在新原型对象建立后创建，因此沿着新原型对象查找，得到 `b.x === 2`

最终输出就是：

```text
1|2
```

#### 延伸说明

需要区分两种操作：

- 修改原型对象属性：会影响所有共享该原型对象的实例
- 直接替换 `Foo.prototype`：只会影响替换之后新创建的实例

### 第 15 题 [多选题]

#### 题目

关于前端存储敏感信息的风险与取舍，哪些说法较合理？

#### 选项

- A. `localStorage` 可被同源脚本读取，发生 XSS 时泄漏风险上升
- B. `HttpOnly Cookie` 降低 JS 读取 Cookie 的风险，但仍需配合 CSRF 防护
- C. 把令牌放在 URL 查询参数中可能进入日志或 Referer 链路
- D. 把令牌放在 DOM 属性中可避免被 JS 读取

#### 答案

A、B、C

#### 解析

这道题的核心在于理解“前端可读存储”和“浏览器自动携带凭证”分别带来的安全权衡。

- A 正确。`localStorage` 对同源 JavaScript 可读，因此一旦页面存在 XSS，攻击脚本就可能直接读取并窃取其中的敏感令牌。
- B 正确。`HttpOnly Cookie` 无法被前端 JS 直接读取，因此能降低 XSS 窃取 Cookie 的风险；但浏览器会自动携带 Cookie，所以仍然需要结合 `SameSite`、CSRF Token 等机制防御 CSRF。
- C 正确。把令牌放进 URL 查询参数，可能被浏览器历史记录、服务器日志、代理日志、监控系统以及 `Referer` 头间接暴露，因此通常不推荐。
- D 错误。DOM 属性同样可以被页面脚本读取，只要前端代码能访问对应节点，就不能把它视为安全隔离区。

#### 延伸说明

前端保存敏感凭证时，常见权衡是：

- 如果重点防止 JS 直接读取，可优先考虑 `HttpOnly Cookie`
- 如果使用 Cookie，则要同步考虑 CSRF 防护
- 如果使用前端可读存储，例如 `localStorage`，就必须更重视 XSS 防护

因此不存在“绝对安全的前端存储位置”，关键在于明确威胁模型并做配套防护。

### 第 16 题 [编程题]

#### 题目

实现一个简化版响应式系统，用来监听对象/数组的数据变化，并在数据变化后自动更新页面上的购物车展示。

页面已经提供两个商品按钮：

- 添加商品 1
- 添加商品 2

点击按钮会修改购物车数据：

- 同一商品重复添加只增加数量
- 购物车列表需要自动刷新到 DOM

本题要求在指定文件中补全两个函数：

- `createProxy(target, handler, basePath)`
- `observe(data, handler)`

#### 功能要求

1. `createProxy(target, handler, basePath)`

- 返回 `target` 的 `Proxy`
- 若 `target` 不是对象或数组，则直接返回原值
- 至少要拦截以下操作：
  - 属性赋值：`proxy[key] = value`
  - 属性删除：`delete proxy[key]`
- 发生拦截后必须：
  - 真实写回原对象 `target`
  - 调用 `handler(change)`

其中 `change` 至少应包含：

```javascript
{
  type: 'set' | 'delete',
  target: <原始对象>,
  key: <当前修改的 key>,
  oldValue: <修改前的值>,
  newValue: <修改后的值>,
  path: <完整路径>
}
```

`path` 拼接规则：

- 若 `basePath` 为空：`String(key)`
- 若 `basePath` 非空：`basePath + '.' + String(key)`

2. `observe(data, handler)`

- 返回一个深度代理后的 `data`
- 对对象/数组任意层级嵌套都能监听
- 任意更深层属性发生 `set/delete` 时，都必须触发监听
- 必须支持循环引用，避免死循环，建议使用 `WeakMap`
- 触发回调时必须使用 microtask 批处理
- `handler` 入参为 `changeList` 数组
- 同一事件循环内的多次变化，只允许触发一次 `handler`
- `changeList` 需要按照变更发生顺序收集多个 `change`
- `change.path` 必须是完整路径，例如：
  - `proxyArr[0].prop1 = 'x'` 时，path 为 `"0.prop1"`
  - `proxy[0].a.b = 1` 时，path 为 `"0.a.b"`

3. 购物车 UI 联动

- 点击 `#btn1` / `#btn2` 后，购物车列表会自动更新
- 同一商品重复点击只增加数量，不新增行
- 每个商品渲染一行，class 必须为 `.cart-item`
- 每行文本格式必须严格为：

```text
商品 1： 3 件
```

注意：

- 使用全角冒号 `：`
- 冒号后有 1 个空格
- 数字后有 1 个空格
- 最后是“件”

#### 严禁修改

以下内容不能修改，否则自动化测试可能失败：

- 按钮 id：`#btn1`、`#btn2`
- 购物车容器 id：`#cart`
- 购物车条目 class：`.cart-item`
- 入口文件名：`/index.html`、`/solution.html`
- 函数签名与函数名：`createProxy(target, handler, basePath)`、`observe(data, handler)`

#### 参考实现思路

整体思路分为两层：

1. `createProxy`

- 只负责单层代理能力
- 通过 `Proxy` 拦截 `get`、`set`、`deleteProperty`
- `set` / `deleteProperty` 里生成标准化 `change`

2. `observe`

- 负责把“单层代理”扩展成“深度监听系统”
- 通过 `WeakMap` 记录已代理对象，避免循环引用导致无限递归
- 通过 `changeList + Promise.resolve().then(...)` 把同一 tick 内的多个变更合并成一次回调

#### 关键代码解析

1. `createProxy` 的路径拼接

```javascript
function joinPath(key) {
  return basePath ? `${basePath}.${String(key)}` : String(key);
}
```

这段逻辑保证了：

- 根层属性如 `count` 的路径是 `"count"`
- 数组第 0 项的 `name` 路径可以是 `"0.name"`
- 更深层嵌套如 `"0.a.b"` 也能自然串联出来

2. `get` 中的延迟深度代理

```javascript
get(currentTarget, key, receiver) {
  const value = Reflect.get(currentTarget, key, receiver);
  if (typeof handler.__getChildProxy === 'function') {
    return handler.__getChildProxy(value, joinPath(key));
  }
  return value;
}
```

这里的核心思想是“按需代理”：

- 不是一开始就递归扫描整个对象树
- 而是在真正访问子对象时，再继续包裹成新的代理

这样实现更轻量，也更适合带循环引用的数据结构。

3. `set` / `deleteProperty` 的标准变更通知

```javascript
handler({
  type: "set",
  target: currentTarget,
  key,
  oldValue,
  newValue: value,
  path,
});
```

这一层的职责是统一上报“发生了什么变化”。

外层 `observe` 并不关心某次变化来自数组索引、普通属性还是深层对象，它只负责把这些变更收集成批。

4. microtask 批处理

```javascript
function scheduleFlush() {
  if (scheduled) return;
  scheduled = true;
  Promise.resolve().then(flushChanges);
}
```

它的意义是：

- 如果同一事件循环里连续发生多次数据变化
- 不要立刻重复调用多次 `handler`
- 而是先把所有变更压入 `changeList`
- 再在 microtask 阶段统一触发一次

这能减少重复渲染，也更符合题目要求。

5. `WeakMap` 解决循环引用与重复代理

```javascript
const proxyCache = new WeakMap();
```

这里用 `WeakMap` 的原因有两个：

- 避免同一个对象反复创建代理
- 支持循环引用结构，避免递归过程中无限套娃

#### 为什么当前实现可满足题目

- `createProxy` 已能正确返回代理对象
- `set/delete` 会真实写入原始对象
- 变更会带上 `type`、`target`、`key`、`oldValue`、`newValue`、`path`
- `observe` 支持对象和数组的深层监听
- `observe` 使用 microtask 合并同一 tick 内的多次修改
- 点击按钮后，购物车 DOM 会自动更新
- 相同商品重复点击只增加数量，不重复新增条目
- `.cart-item` 文案格式已修正为断言要求的精确形式

#### 复杂度分析

- 单次 `set` / `delete` 代理拦截：平均可视为 `O(1)`
- 路径拼接：与路径深度相关，可视为 `O(d)`

- 购物车渲染：与购物车商品种类数 `n` 成正比，为 `O(n)`

### 第 17 题 [编程题]

#### 题目

小苯面前有 `n` 个纯白色数字，其中第 `i` 个数字的值为 `a_i`。

小苯拿到了 `m` 个区间 `[l_j, r_j]`，每个区间都用来描述一个涂色操作，将 `[l_j, r_j]` 这段区间中的所有数字都涂成红色。小苯可以任意选择其中一些操作执行，当然也可以一个都不执行，也可以全部执行。

小苯只能通过上述方式涂红数字，不能随意涂色。最终小苯的得分是所有红色数字的和。

现在小苯想知道，如果限制所选区间两两不相交，他的得分最高能有多少。

#### 输入描述

每个测试文件均包含多组测试数据。

第一行输入一个整数 `T`，表示数据组数。

对于每组测试数据：

- 第一行输入两个正整数 `n, m`，表示数字个数和可选操作个数
- 第二行输入 `n` 个整数 `a_i`，表示每个数字的值
- 接下来 `m` 行，每行输入两个正整数 `l_j, r_j`，描述一个操作区间

额外保证：

- `1 <= T <= 10^5`
- `1 <= n, m <= 3 * 10^5`
- `-10^9 <= a_i <= 10^9`
- `1 <= l_j <= r_j <= n`
- 单个测试文件中，所有测试数据的 `n` 之和不超过 `3 * 10^5`
- 单个测试文件中，所有测试数据的 `m` 之和不超过 `3 * 10^5`

#### 输出描述

对于每组测试数据，在单独的一行输出一个整数，表示小苯的最大得分。

#### 核心结论

把每个区间 `[l, r]` 看成一个“带权区间”：

- 区间权重 = `sum(a[l..r])`
- 目标 = 选择若干两两不相交区间，使总权重最大

这就是标准的“带权区间调度”问题。

#### 解题思路

1. 前缀和预处理区间权重

设：

```text
prefix[i] = a[1] + a[2] + ... + a[i]
```

那么任意区间 `[l, r]` 的权重都可以在 `O(1)` 时间求出：

```text
weight(l, r) = prefix[r] - prefix[l - 1]
```

2. 将所有区间按右端点排序

把每个区间记为：

```text
(r, l, weight)
```

按 `r` 从小到大排序后，问题就变成：

- 枚举“最后一个选的区间”
- 找到它前面最后一个与它不相交的区间

3. 动态规划

设 `dp[i]` 表示“排好序后的前 `i` 个区间里，最多能取得的最大总得分”。

对于第 `i` 个区间，只有两种选择：

- 不选它：`dp[i - 1]`
- 选它：`dp[prev] + weight`

其中 `prev` 表示最后一个满足“右端点 < 当前区间左端点”的区间数量。

转移方程：

```text
dp[i] = max(dp[i - 1], dp[prev] + weight)
```

4. 二分查找前驱区间

因为区间已经按右端点排序，所以可以把所有右端点单独提出来形成数组 `ends`。

对当前区间左端点 `l`，我们希望找到：

```text
最后一个 end < l 的位置
```

等价于：

```text
prev = lower_bound(ends, l)
```

这里 `prev` 恰好就是可以安全接在当前区间前面的区间数量。

#### 为什么现有 `2.py` 是正确的

[2.py](./2.py) 的实现正是上述做法：

- 用前缀和计算每个区间的权重
- 按右端点排序
- 用 `bisect_left` 找前驱
- 用一维 `dp` 做带权区间调度

对应关键逻辑如下：

```python
weight = prefix[right] - prefix[left - 1]
intervals.append((right, left, weight))
intervals.sort()
prev = bisect.bisect_left(ends, left)
dp[i] = max(dp[i - 1], dp[prev] + weight)
```

### 第 18 题 [编程题]

#### 题目

给定一个长度为 `n` 的数字序列。每次如果存在一对相邻且相等的数字，就可以将这一对数字同时消除，剩余部分重新拼接后继续进行，直到无法继续消除为止。

现在允许你在原序列中的任意位置额外插入一个数字，插入的数字数值也可以自行选择。你需要让最终能够消除的对数尽量多，输出最多能消除多少对数字。

根据现有代码与题图信息，可以确定：

- 输入包含多组测试数据
- 每组先给出 `n`，再给出长度为 `n` 的整数序列
- 输出每组数据的最大可消除对数

#### 核心结论

这题可以拆成两段来思考：

1. 先把原序列中“无论是否插入新数字都一定会发生”的相邻消除全部做完
2. 对剩下无法继续消除的序列，考虑插入一个数字以后，最多还能额外触发多少次连锁消除

第一段可以用栈线性完成，第二段本质上是在剩余序列里寻找最长的奇数长度回文结构。

#### 为什么可以先做贪心消除

从左到右扫描原数组，用栈维护当前还没有被消掉的数字：

- 若当前数字和栈顶相等，就说明形成了一对相邻相等数字，可以立即消掉
- 否则把当前数字压栈

这个过程结束后：

- `base` 表示原序列自身已经能够消除的对数
- `stack` 表示所有必然消除完成后剩下的“稳定序列”

这里的栈化简结果可以理解为该消除规则下的唯一稳定形态，因此先把“原本一定会发生的消除”全部做完，并不会影响“额外插入一个数字后还能继续触发多少次消除”的最优答案。

稳定序列有一个重要性质：

- 相邻元素一定不相等

所以后续如果还想继续消除，唯一办法就是通过插入一个数字，去连接两边本来被隔开的相同数字，从而触发一条向外扩展的连锁反应。

#### 插入一次后，额外收益来自奇回文半径

假设稳定序列中存在一段：

```text
a b c b a
```

如果在中间位置插入一个 `c`，就会先变成：

```text
a b c c b a
```

先消掉中间这一对 `c c` 之后，左右两侧又变成相邻且相等的 `b b`，继续消除；再往外还有 `a a`，于是可以连续消掉 3 对。

这说明：

- 只要稳定序列以某个位置为中心，左右两边对称相等
- 在中心插入一个与中心值相同的数字
- 就能把这一整段奇数长度回文全部转化成连续消除

因此答案就是：

```text
最终答案 = base + best
```

其中：

- `base` 是原序列本来就能消掉的对数
- `best` 是稳定序列中最大的 `d1[i]`

要特别注意，这里的 `d1[i]` 不是“去掉中心后的扩展层数”，而是“包含中心本身的奇回文半径”：

- `d1[i] = 1` 表示只取中心一个数，此时仍然可以在它旁边插入同值数字，额外消掉 `1` 对
- `d1[i] = 2` 表示形如 `a b a`，插入一个 `b` 后可额外消掉 `2` 对
- `d1[i] = 3` 表示形如 `a b c b a`，插入一个 `c` 后可额外消掉 `3` 对

所以 `best` 可以直接作为“插入一次后最多还能额外消除多少对数字”，最终答案直接写成 `base + best` 即可，不需要再做 `-1` 或其他换算。

#### 解题思路

1. 用栈消掉原序列中所有必然发生的相邻相等对，得到 `base` 和剩余序列 `stack`
2. 如果 `stack` 为空，说明原序列已经能全部消完，答案就是 `base`
3. 否则在 `stack` 上计算每个位置作为中心时的最长奇回文半径
4. 设最大半径为 `best`，则答案为 `base + best`

这里使用的是 Manacher 的奇回文版本，能在线性时间内求出所有中心的最大扩展半径。

#### 关键转移

设 `d1[i]` 表示以 `i` 为中心的最长奇回文半径，包含中心自己，最小值为 `1`。

例如：

```text
stack = [1, 2, 3, 2, 1]
```

则中间位置 `3` 的 `d1[i] = 3`，表示可以贡献 3 对消除：

- 插入一个 `3`
- 触发 `3 3`
- 再触发 `2 2`
- 再触发 `1 1`

Manacher 的扩展逻辑为：

```text
如果 i 在当前最右回文覆盖范围外，直接从 1 开始扩展
否则先利用对称位置的结果做下界，再继续尝试扩展
```

最终取所有 `d1[i]` 的最大值即可。

#### 复杂度分析

对每组测试数据：

- 栈消除：`O(n)`
- 奇回文半径计算：`O(n)`

总时间复杂度：

```text
O(n)
```

空间复杂度：

```text
O(n)
```
